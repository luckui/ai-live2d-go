# Writing Implementation Plans

**Use when:**
- Have spec or requirements for multi-step task
- Complex feature implementation
- Uncertain how to break down task
- Need detailed implementation plan

**Core principle:** A good plan makes implementation obvious. If someone has to guess, the plan is incomplete.

---

## When to Use

**Always use before:**
- Implementing multi-step features
- Breaking down complex requirements
- Starting any non-trivial implementation

**Don't skip when:**
- Feature seems simple (assumptions cause bugs)
- Plan to implement it yourself (future you needs guidance)
- Working alone (documentation matters)

---

## Bite-Sized Task Granularity

**Each task = 2-5 minutes of focused work.**

Every step is one action:
- "Write the failing test" — step
- "Run it to make sure it fails" — step
- "Implement the minimal code to make the test pass" — step
- "Run the tests and make sure they pass" — step
- "Commit" — step

**Too big:**
```markdown
### Task 1: Build authentication system
[50 lines of code across 5 files]
```

**Right size:**
```markdown
### Task 1: Create User model with email field
[10 lines, 1 file]

### Task 2: Add password_hash field to User
[8 lines, 1 file]

### Task 3: Create password hashing utility
[15 lines, 1 file]
```

---

## Plan Document Structure

### Header (Required)

Every plan MUST start with:

```markdown
# [Feature Name] Implementation Plan

**Goal:** [One sentence describing what this builds]

**Approach:** [2-3 sentences about approach]

**Tech Stack:** [Key technologies/libraries]

---
```

---

### Task Structure

Each task follows this format:

````markdown
### Task N: [Descriptive Name]

**Objective:** What this task accomplishes (one sentence)

**Files:**
- Create: `exact/path/to/new_file.ts`
- Modify: `exact/path/to/existing.ts` (line numbers if known)
- Test: `tests/path/to/test_file.test.ts`

**Step 1: Write failing test**

```typescript
// tests/feature.test.ts
test('describe expected behavior', () => {
  const result = newFunction();
  expect(result).toBe(expected);
});
```

**Step 2: Run test see it fail**

```bash
npm test -- --testNamePattern="describe expected behavior"
```

**Expected output:** Test fails with error: `newFunction is not defined`

**Step 3: Implement minimal code**

```typescript
// src/feature.ts
export function newFunction() {
  return expected;
}
```

**Step 4: Verify test passes**

```bash
npm test
```

**Expected output:** All tests pass

**Step 5: Commit**

```bash
git add src/feature.ts tests/feature.test.ts
git commit -m "feat: add newFunction"
```

---
````

## Principles

### DRY (Don't Repeat Yourself)

**Bad:** Copy-paste validation in 3 places
**Good:** Extract validation function, use everywhere

### YAGNI (You Aren't Gonna Need It)

**Bad:** Add "flexibility" for future requirements
**Good:** Implement only what's needed now

```typescript
// Bad — YAGNI violation
class User {
  constructor(name: string, email: string) {
    this.name = name;
    this.email = email;
    this.preferences = {};  // Not needed yet!
    this.metadata = {};     // Not needed yet!
  }
}

// Good — YAGNI
class User {
  constructor(name: string, email: string) {
    this.name = name;
    this.email = email;
  }
}
```

### TDD (Test-Driven Development)

Every task that produces code should include the full TDD cycle:
1. Write failing test
2. Run to verify failure
3. Write minimal code
4. Run to verify pass

See "Test-Driven Development" manual for details.

### Frequent Commits

Commit after every task:
```bash
git add [files]
git commit -m "type: description"
```

**Commit types:**
- `feat:` New feature
- `fix:` Bug fix
- `refactor:` Refactoring
- `test:` Add tests
- `docs:` Documentation

## Common Mistakes

### Vague Tasks

**Bad:** "Add authentication"
**Good:** "Create User model with email and password_hash fields"

### Incomplete Code

**Bad:** "Step 1: Add validation function"
**Good:** "Step 1: Add validation function" followed by the complete function code

### Missing Verification

**Bad:** "Step 3: Test it works"
**Good:** "Step 3: Run `npm test`, expected: 3 passed"

### Missing File Paths

**Bad:** "Create the model file"
**Good:** "Create: `src/models/user.ts`"

## Task Breakdown Example

### Bad plan:
```markdown
## Implement user login

1. Create login form
2. Add validation
3. Connect backend
4. Handle errors
```

### Good plan:
```markdown
## Implement user login

### Task 1: Create login form component skeleton
- Create: `src/components/LoginForm.tsx`
- Empty component, returns `<form>`
- 2 minutes

### Task 2: Add email input field
- Modify: `src/components/LoginForm.tsx`
- Add controlled email input
- Write test: form displays email field
- 3 minutes

### Task 3: Add password input field
- Modify: `src/components/LoginForm.tsx`
- Add controlled password input (type="password")
- Write test: form displays password field with type password
- 3 minutes

### Task 4: Add submit button
- Modify: `src/components/LoginForm.tsx`
- Add submit button, disabled by default
- Write test: button enabled when email and password filled
- 4 minutes

### Task 5: Add email format validation
- Create: `src/utils/validation.ts`
- Function `isValidEmail(email: string): boolean`
- Write test: valid/invalid email formats
- TDD cycle
- 5 minutes

### Task 6: Integrate email validation into form
- Modify: `src/components/LoginForm.tsx`
- Import `isValidEmail`
- Show error message when email invalid
- Write test: invalid email shows error
- 4 minutes

[Continue...]
```

## Plan Checklist

Before plan is complete, check:

- [ ] Each task is 2-5 minutes
- [ ] Includes exact file paths
- [ ] Includes complete code examples (copy-pasteable)
- [ ] Includes exact commands with expected output
- [ ] Includes verification steps
- [ ] Follows DRY, YAGNI, TDD
- [ ] Has commit after each task
- [ ] Tasks in logical order
- [ ] No vague steps ("configure it", "make it work")

## Using todo tool for tracking

When implementing plan, use `todo` tool to manage progress:

```typescript
// Write task list
todo.write([
  { id: 1, title: 'Create login form component skeleton', status: 'pending' },
  { id: 2, title: 'Add email input field', status: 'pending' },
  { id: 3, title: 'Add password input field', status: 'pending' },
  // ...
]);

// Start task
todo.write([
  { id: 1, title: 'Create login form component skeleton', status: 'in_progress' },
  // ...
]);

// Complete task
todo.write([
  { id: 1, title: 'Create login form component skeleton', status: 'completed' },
  { id: 2, title: 'Add email input field', status: 'in_progress' },
  // ...
]);
```

**Rules:**
- List order = priority
- Only 1 `in_progress` at a time
- Mark `completed` immediately after finishing
